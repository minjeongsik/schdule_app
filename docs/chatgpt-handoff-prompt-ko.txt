당신은 지도 기반 일정 관리 MVP 프로젝트를 이어서 작업하는 역할입니다.

아래 현재 상태를 먼저 읽고, 새로 갈아엎는 방향이 아니라 지금 코드베이스를 점진적으로 발전시키는 방향으로 답변하세요.

중요 원칙

1. MVP 기준을 유지합니다.
2. 과설계하지 않습니다.
3. 기존 구조를 가능한 한 유지합니다.
4. 작은 변경과 작은 커밋 단위로 접근합니다.
5. `todo` 코드는 별도 요청이 없는 한 레거시로 간주합니다.

현재 기술 스택

- 백엔드: Express + TypeScript + Prisma + SQLite
- 프론트엔드: React + TypeScript + Vite
- 데이터 패칭: React Query
- 검증: Zod

현재 도메인 규칙

- Place는 user scoped
- Appointment는 여러 Route candidate를 가짐
- Route candidate는 여러 Waypoint를 가짐
- 프론트 에러는 `field / form / page`로 분리됨

현재 구현 상태

- Places CRUD 구현 완료
- Appointments CRUD 구현 완료
- Route candidate 생성 / 선택 / 수정 / 삭제 구현 완료
- Waypoint 저장 지원
- 선택된 route를 UI에서 확인 가능
- route candidate 관리는 appointment 대시보드 안에서 처리
- `todo` 런타임 라우트는 연결되어 있지 않음

현재 주요 API

```http
GET    /api/appointments?userId=demo-user
GET    /api/appointments/:id
POST   /api/appointments
PATCH  /api/appointments/:id
DELETE /api/appointments/:id

POST   /api/appointments/:id/routes
PATCH  /api/appointments/:id/routes/:routeId
PATCH  /api/appointments/:id/routes/:routeId/select
DELETE /api/appointments/:id/routes/:routeId

GET    /api/places?userId=demo-user
POST   /api/places
PATCH  /api/places/:id
DELETE /api/places/:id?userId=demo-user
```

현재 Prisma 관련 핵심 사실

- `Place.userId` 존재
- route candidate 모델 이름은 아직 `SavedRoute`를 사용
- `Waypoint` 모델 존재

중요한 로컬 환경 주의사항

- 올바른 SQLite 경로는 `file:./prisma/dev.db`
- `server/.env`, `server/.env.example`는 반드시 아래 값이어야 함

```env
DATABASE_URL="file:./prisma/dev.db"
```

- 과거에 `file:./dev.db`를 써서 잘못된 DB를 보는 버그가 있었음
- env 수정 후 `Internal server error`가 보이면 서버 프로세스를 재시작해야 함
- `GET /api/health`의 `databaseUrl`이 `/server/prisma/dev.db`로 끝나야 정상

레거시 상태

- `todo` 관련 파일은 아직 저장소에 남아 있음
- 현재 런타임은 scheduler 모듈 기준으로 동작
- `todo`는 참고용 레거시로 보고, 특별한 요청이 없으면 건드리지 않음

현재 검증 상태

- server lint/build 통과
- client lint/test/build 통과
- 로컬 DB 경로 문제 수정 및 reseed 완료

다음 어시스턴트가 지켜야 할 방향

- 큰 구조 개편을 제안하지 말 것
- 추상화만 위한 repository layer를 억지로 만들지 말 것
- 명확한 필요가 생기기 전까지 프론트를 과도하게 분리하지 말 것
- scheduler 동작, 검증, 테스트, UX를 점진적으로 개선할 것
